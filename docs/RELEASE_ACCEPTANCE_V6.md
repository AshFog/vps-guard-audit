# v6 发布验收清单

本清单用于决定 v6 候选版本能否发布。自动化测试通过只是必要条件，不能替代真实 VPS 的 SSH、防火墙、systemd 和厂商控制台验收。

## 一、自动化发行矩阵

从 GitHub Actions 手动运行 `v6 发行验收`，或为 v6 分支创建 Pull Request。以下六个环境必须全部通过：

- Ubuntu 26.04、24.04、22.04 LTS；
- Debian 13、12、11。

每个环境会在无网络的一次性容器中完成：

1. 核对18项注册表动作都被测试脚本真实调用；
2. 运行权限、SSH、系统、连接保护、防火墙、Fail2ban、网络策略和服务状态故障测试；
3. 全新安装当前候选版本并运行 `vpsga doctor`；
4. 生成 Schema 2.0 快速审计 JSON 和中文只读加固计划；
5. 安装真实 `v5.0.0` tag，再升级到当前 v6 候选版本；
6. 确认旧 release 目录与 `/var/lib/vps-guard-audit` 状态保留；
7. 复核安装所有者、权限、`MANIFEST.sha256` 和升级后只读审计。

测试脚本 `tests/test-install-lifecycle.sh` 会删除一次性系统内的 VPSGA 安装目录。它只有在 root、非 SSH 会话且显式设置 `VPSGA_E2E_DISPOSABLE=YES` 时才会运行，禁止在生产 VPS 上执行。

## 二、真实 VM 准备

至少准备一台全新 Ubuntu LTS VM 和一台全新 Debian VM。建议优先使用 Ubuntu 24.04 与 Debian 13，再按发布风险抽查其余受支持版本。

每台 VM 都必须具备：

- 可用的云厂商网页控制台、VNC 或救援模式；
- 执行前快照；
- root 当前 SSH 会话；
- 一个非 root sudo 管理员及已实际登录成功的独立 SSH 密钥；
- 第二个 SSH 终端；
- 明确记录的 SSH、HTTP/HTTPS、面板、代理、VPN 和 Docker 发布端口。

不得在唯一一台生产服务器上完成首次验收。

## 三、安装与审计

1. 从候选分支执行本地 `install.sh`，确认 `vpsga --version` 与候选版本一致。
2. 运行 `vpsga doctor`，不得出现权限、所有者、符号链接或哈希问题。
3. 分别运行 quick、standard 和 deep 审计；退出码只允许为 0、1 或 2。
4. 生成 TXT、AI 脱敏报告和 JSON，检查文件权限为 `0600`，JSON Schema 为 2.0。
5. 连续运行两次审计，确认历史比较能识别新增、已解决和等级变化。
6. 运行 `vpsga plan`，确认计划阶段没有修改系统。

## 四、10项常规加固

在快照或可恢复测试机上逐项执行 `HARD-1001` 至 `HARD-1010`：

1. 执行前记录目标文件权限、内容、服务状态和相关 sysctl 值；
2. 输入 `APPLY` 后确认动作成功、事务状态为 `committed`；
3. 重新运行审计，确认对应发现闭环；
4. 对每种动作类型至少选择一项执行 `vpsga rollback <事务编号>`；
5. 确认文件、权限、SSH reload、sysctl 运行时值和服务配置恢复；
6. 人工制造一次验证失败，确认事务自动恢复且状态不是 `committed`。

自动安全更新测试允许回滚后保留新安装的软件包，但配置必须恢复。任何 SSH 动作都必须保留当前连接直到复检完成。

## 五、8项连接敏感加固

先运行：

```bash
vpsga connection-check
vpsga firewall-plan
vpsga workload-plan
```

逐项验收 `HARD-2001` 至 `HARD-2008`，每项都执行两条路径：

### 确认路径

1. 原终端启动动作并确认控制台可用；
2. 选择已验证的备用 sudo 密钥管理员；
3. 确认 systemd 临时回滚 timer 已建立；
4. 从第二个独立 SSH 会话执行给出的确认命令；
5. 返回原终端提交事务；
6. 再开一个新 SSH 会话并复检业务。

### 超时路径

1. 再次从干净快照启动同类动作；
2. 不在第二终端确认，等待5分钟 timer 触发；
3. 确认事务变为 `rolled_back`；
4. 确认原 SSH 配置、UFW 规则、Fail2ban 状态、sysctl 值或服务状态恢复；
5. 确认原 SSH 会话与新登录均可用。

还必须验证：原终端不能伪造第二终端确认，错误管理员、错误端口和过期令牌均被拒绝；等待期间若人工修改同一配置，自动回滚应拒绝覆盖并提示使用控制台处理。

## 六、业务回归

完成敏感动作后逐项复核实际业务：

- SSH 密钥登录、sudo 和紧急控制台；
- 网站 80/443、管理面板及自定义 SSH 端口；
- Docker 发布端口及 `DOCKER-USER` 链；
- WireGuard/OpenVPN、代理、软路由与 IPv4/IPv6 转发；
- SSH `-L/-R/-D`、VS Code Remote、跳板机和数据库隧道；
- CUPS/Avahi 确实不再需要时才允许停用。

任何一项用途无法确认，都应保留原状态，不得为了“全部通过”而强行执行动作。

## 七、升级与发布判定

1. 在另一台快照 VM 安装 v5.0.0，生成至少一份历史报告；
2. 升级到 v6 候选版本；
3. 确认 `current` 指向 v6、v5 release 保留、历史与加固事务未丢失；
4. 运行 `vpsga doctor` 和 standard 审计；
5. 验证现有 v5 配置档案仍可读取；
6. 在 GitHub 核对常规 CI 与六系统发行矩阵均为绿色；
7. 记录 VM 镜像、系统版本、测试时间、测试人、失败与修复提交。

只有以下条件全部满足才可发布 v6.0.0：

- 常规 CI 通过；
- 六系统自动化发行矩阵通过；
- Ubuntu 与 Debian 真实 VM 的确认路径和超时路径通过；
- v5 至 v6 真实升级通过；
- 没有未解释的断连、服务中断、状态丢失或回滚失败。
