# 连接敏感加固与失联恢复

连接敏感加固不是“更高级的一键选项”。它修改 SSH、防火墙、转发或系统服务，错误配置可能让服务器仍在运行，但你无法再通过网络进入。

## 操作前必须准备

1. 登录 VPS 商家后台，确认网页控制台、VNC、串口控制台或救援模式可以打开。
2. 保留当前 SSH 窗口，不要在验证完成前退出。
3. 打开第二个终端，确认当前登录方式和端口能够建立新连接。
4. 记录 SSH 端口、服务器 IP、需要开放的 TCP/UDP 端口，以及正在运行的 Docker/代理/VPN 服务。
5. 确认至少有一个可以登录并使用 sudo 的管理账户。

## 工具已经建立的保护框架

v6.0.0-dev.7 已使用以下统一流程开放 `HARD-2001` 至 `HARD-2008`：

1. 记录相关文件、权限、所有者和哈希。
2. 创建只允许 root 读取的恢复包。
3. 设置独立的延时自动回滚任务。
4. 用临时文件生成新配置并进行语法检查。
5. 应用后保留当前连接，生成10分钟有效且绑定本次事务的确认令牌。
6. 第二终端必须由已验证的非 root sudo 管理员登录同一服务器入口，并运行 `vpsga connection-confirm TOKEN`。
7. 只有不同 SSH 会话验证成功，才提交事务并取消自动回滚。
8. 重新运行对应检查；验证失败则立即恢复。

可以先运行以下命令检查当前主机是否具备这些前置条件：

```bash
vpsga connection-check
```

涉及 UFW 前，另行运行只读端口计划并人工核对：

```bash
vpsga firewall-plan
```

涉及 SSH 隧道、IP 转发、IPv6、Docker/VPN 或候选服务前，运行只读业务用途检查：

```bash
vpsga workload-plan
```

工具不会把“我有控制台”当作能够自动检测的事实。正式执行连接敏感动作时，用户仍必须明确确认 VPS 商家的网页控制台、VNC、串口控制台或救援模式确实可用。

## 已经失联怎么办

通过 VPS 商家控制台进入系统后，先确认是哪一层阻断：

```bash
systemctl status ssh
sshd -t
ss -lntp
ufw status numbered
fail2ban-client status sshd
```

不要先重启服务器。重启可能关闭仍然存活的 SSH 会话，也可能让尚未验证的配置正式生效。应先恢复最近一次 VPS Guard Audit 备份，再验证 SSH 配置并重新加载服务。

网络与服务状态也会记录在事务中。若等待确认期间又有人改变 sysctl 或重新启动候选服务，自动回滚会拒绝覆盖后来变更，并要求通过 VPS 控制台人工检查。
